## Slice 1.1 — 디자인 시스템 토큰

### 배경
현재 `src/app.css`는 색상 토큰만 10개 정의하고, 타이포그래피/간격/라운딩/밀도 토큰은 없다.
각 컴포넌트가 Tailwind 유틸(`text-sm`, `p-4` 등)을 하드코딩하여 일관성이 흔들린다.

### 목표
- Tailwind 4 `@theme` 안에서 참조 가능한 토큰 집합 정의
- **밀도(density)** 개념 도입 → 컴팩트 모드의 기반 (Slice 1.2에서 활용)
- 노션 라이트 톤을 참고한 색상 리밸런싱 (기존 #0a0a0a 기반은 너무 짙음)

### 토큰 카테고리

**Color**
- Surface (base/1/2/3) — 배경 레이어링
- Text (primary/secondary/muted/disabled)
- Border (default/strong/accent)
- Accent (blue 계열)
- Semantic (success/warning/danger/info)

**Typography**
- Font family: Sans(기본) + Mono(코드)
- Font size scale: xs(11) / sm(12) / base(13) / md(14) / lg(16) / xl(18) / 2xl(22) / 3xl(28)
  - 노션 라이트는 14px 기준, GitHub 마크다운은 16px 기준 → 컨텍스트별 선택
- Line height: tight(1.3) / normal(1.5) / relaxed(1.7)

**Spacing (density-aware)**
- `--space-1` ~ `--space-8` CSS 변수
- 컴팩트 모드에서 모든 값이 75% 비율로 축소됨 (Slice 1.2에서 구현)

**Radius / Shadow**
- `--radius-sm/md/lg/full`
- `--shadow-xs/sm/md`

### TDD 테스트 대상
디자인 토큰은 기본적으로 선언적이라 전통적 유닛 테스트가 약하다. 대신:
- Svelte 컴포넌트 시각 회귀 테스트는 MVP 범위 외 (수동 검증)
- 토큰 네이밍 일관성은 eslint/stylelint 룰 없이 리뷰로 보장
- **검증 방법**: 모든 페이지를 한 번씩 돌면서 하드코딩된 hex/픽셀 값을 토큰으로 치환 확인

### 구현 단계
1. `src/app.css`에 전체 토큰 추가 (@theme)
2. 각 페이지/컴포넌트에서 하드코딩 제거 (Grep으로 `#[0-9a-f]{6}`, `\dpx` 검색 후 치환)
3. 다크 테마 기준으로 색상 리밸런싱 (surface를 #0e0e0e → #111111 등 살짝 밝게)

## Slice 1.2 — 컴팩트 모드

### 배경
사용자는 가벼운 노션 스타일 밀도를 원한다. 설정에서 기본/컴팩트를 전환하면 전체 앱 간격/글꼴이 축소.

### Spec 스케치

**백엔드 (`AppConfig` 확장)**
- 신규 필드: `pub density: Density` (enum: `Regular`, `Compact`) — `serde(default)` migration-safe
- `update_config` patch에 `density` 추가

**프론트엔드 전역 적용**
- `<html data-density="compact">` 속성을 layout에서 읽어서 설정
- `app.css`의 spacing 토큰이 `[data-density="compact"]` 하위에서 축소값으로 오버라이드
  ```css
  :root { --space-base: 16px; }
  [data-density="compact"] { --space-base: 12px; }
  ```
- 폰트 크기도 같은 방식 (--text-base가 14px → 12px)

**UI**
- `/settings` 페이지에 "밀도" 섹션 (라디오: 기본/컴팩트)
- 저장 즉시 전역 반영 (재시작 불필요)

### TDD 테스트 대상
- `AppConfig` serde default: 구 config에 `density` 필드 없을 때 Regular
- `update_config(patch)` 부분 patch에 density 머지
- density enum serialize/deserialize roundtrip

### 검증
- 컴팩트 전환 → 대시보드 카드 간격 축소, 사이드바 밀도 증가, 본문 폰트 축소
- 설정 재로드 시 유지

## Slice 1.3 — GitHub 스타일 마크다운 렌더러

### 배경
현재 `vault/renderer.rs`는 pulldown-cmark 기본 출력을 그대로 사용. 코드 하이라이팅 없음, 테이블/체크박스 스타일 미흡.
사용자는 GitHub 마크다운 수준의 가독성 + 수식/다이어그램/DB 스키마 렌더링을 원함.

### 결정: 프론트엔드 스타일/렌더 레이어 접근
Rust 렌더러 교체는 비용이 크고 테스트 자산을 버리게 됨. 대신:
1. **Rust 렌더러는 유지** — pulldown-cmark에 GFM 옵션(테이블, 체크박스, 취소선) 추가만 활성화
2. **프론트엔드에서 CSS + 하이라이팅 + 수식/다이어그램 렌더링 오버레이**

### 범위
| 기능 | 구문 | 라이브러리 | 번들 크기 |
|------|------|---------|---------|
| GFM 기본 (테이블, 체크박스, 취소선) | 마크다운 표준 | pulldown-cmark | - |
| GitHub 시각 스타일 | `.markdown-body` 래퍼 | github-markdown-css | ~30KB |
| 코드 하이라이팅 | ` ```lang ` | Prism.js (주요 언어) | ~50KB |
| 수식 | `$...$` / `$$...$$` | KaTeX | ~300KB |
| 일반 다이어그램 | ` ```mermaid ` | Mermaid | ~500KB |
| DB 스키마 | ` ```dbml ` | @softwaretechnik/dbml-renderer | ~400KB |

### 구현

**Rust 측 (작은 변경)**
- `pulldown_cmark::Options`에 GFM 확장 옵션 모두 활성화
  - `ENABLE_TABLES`, `ENABLE_FOOTNOTES`, `ENABLE_STRIKETHROUGH`, `ENABLE_TASKLISTS`, `ENABLE_SMART_PUNCTUATION`
- 코드 블록에 `language-xxx` 클래스가 붙는지 확인 (pulldown-cmark 기본 지원)
- 수식(`$...$`/`$$...$$`)은 pulldown-cmark가 자체 처리하지 않으므로 **그대로 통과** → 프론트에서 후처리

**프론트엔드 측 — 뷰어 렌더 파이프라인**
```
HTML (from Rust) → .markdown-body 래퍼
  → Prism.highlightAllUnder (non-mermaid, non-dbml 코드 블록)
  → mermaid.run() (language-mermaid 코드 블록 → SVG 치환)
  → dbmlRenderer (language-dbml 코드 블록 → SVG 치환)
  → renderMathInElement (KaTeX — $...$/$$...$$ 치환)
```

렌더 순서 중요:
1. **KaTeX 먼저** 실행하면 `$...$` 속 달러 기호를 수식 노드로 치환 → 후속 처리 안전
2. **Prism 다음** — 코드 블록 토큰화 (mermaid/dbml 블록은 제외)
3. **Mermaid/DBML 마지막** — 원본 코드 블록을 SVG로 치환

**의존성 추가**
- `katex` — 수식 렌더
- `mermaid` — 다이어그램 렌더
- `@softwaretechnik/dbml-renderer` — DBML → SVG
- `prismjs` — 코드 하이라이팅
- `github-markdown-css` — GitHub 스타일

**Lazy-loading (번들 최적화)**
- 뷰어 진입 시점에 동적 import:
  ```ts
  const { default: katex } = await import("katex");
  ```
- 노트 내용에 실제로 수식/mermaid/dbml 블록이 있을 때만 해당 라이브러리 로드

**CSS 격리**
- `.markdown-body` 스코프 내에서만 GitHub 스타일 적용
- 디자인 토큰의 색상/간격은 `.markdown-body {}` 오버라이드로 통합
- 코드 블록 배경은 토큰(`--color-surface-2`) 사용하여 앱 전체 톤과 일치

### TDD 테스트 대상 (Rust)
- GFM 옵션 활성화 후 기존 14개 렌더러 테스트 회귀 없음
- 테이블/체크박스/취소선 렌더링 신규 테스트 (각 1개)
- 수식 구문(`$...$`, `$$...$$`)이 HTML에 원형 그대로 통과되는지 (프론트 후처리를 위함)
- mermaid/dbml 코드 블록이 `language-mermaid`/`language-dbml` 클래스로 보존되는지

### 검증 (프론트엔드 수동)
- 샘플 노트에 각 요소 포함:
  - 코드 블록(rust/python/bash/ts) → 언어별 하이라이팅
  - `$E=mc^2$` 인라인 수식 + `$$\int_0^1 x^2 dx$$` 블록 수식
  - `sequenceDiagram`, `flowchart`, `erDiagram` (Mermaid)
  - `Table users { ... }` (DBML)
  - 테이블/체크박스/취소선/각주
- wikilink 여전히 동작 (Rust 측에서 이미 HTML 앵커로 변환됨)
- 컴팩트 모드 전환 시 본문 폰트/간격 축소 반영

### 오류 처리
- Mermaid/DBML 파싱 실패 시: 원본 코드 블록을 보존하고 "Failed to render" 주석 오버레이
- KaTeX 파싱 실패 시: 원본 `$...$` 문자열 유지
- 렌더 전체가 깨지지 않게 각 후처리기를 try-catch

### 열린 결정
- **하이라이팅 라이브러리**: Prism (가볍고 동적), shiki (VS Code 테마, 더 정확) → MVP는 Prism으로 시작
- **링크 색상**: GitHub 라이트 파란 vs 디자인 토큰 accent → accent로 통일
- **Mermaid 테마**: dark/forest/default 중 선택 → 디자인 토큰과 어울리는 `dark` 시작
- **KaTeX 구분자**: `$...$` + `$$...$$` 표준 vs 백슬래시 `\(...\)` 추가 지원 → MVP는 `$` 표준만

## Slice 1.5 — 라이트/다크 테마 스위칭

### 배경
Slice 1.1의 토큰은 다크 팔레트만 정의. Slice 1.3에서 `html[data-theme]` 분기 훅을
심어놨지만(`github-markdown-dark.css`, Mermaid 테마), 실제 라이트 팔레트/UI 토글은 미구현.

"전반적 테마를 밝게 바꿀 예정"이라는 방향이 있고, 접근성/선호 다양성 대응이 필요.

### 목표
- 토큰 단에서 라이트/다크 두 팔레트 **완전 분리**
- 앱 어디서나 1초 안에 전환 가능한 토글
- OS 다크 모드 연동(`system`) 옵션
- 설정 persist (`AppConfig.theme`)
- Mermaid/github-markdown/Prism 토큰까지 일관되게 반영

### 토큰 설계

**분리 전략**: 루트 `@theme`은 공통 구조 변수(spacing/radius/font)만 두고, 색상 토큰은
`[data-theme="dark"]`/`[data-theme="light"]` 블록에 각각 정의.

```css
@theme {
  /* 구조 토큰만 — 색상 제외 */
  --spacing: 4px;
  --radius-md: 6px;
  --font-sans: ...;
}

[data-theme="dark"] {
  --color-surface-0: #0e0e0e;
  --color-fg: #f5f5f5;
  /* ... 11개 컬러 토큰 다크 값 */
}

[data-theme="light"] {
  --color-surface-0: #ffffff;
  --color-fg: #1f1f1f;
  /* ... 11개 컬러 토큰 라이트 값 */
}
```

라이트 팔레트는 **노션 라이트**를 참조점으로 잡되, 대비(WCAG AA)와 분위기(Karpathy 위키 톤)
사이의 균형을 별도 디자인 단계에서 확정. `/design-consultation`으로 팔레트 프리뷰 페이지 생성.

### 런타임 분기 대상

| 대상 | 메커니즘 |
|------|---------|
| 토큰 색상 | `[data-theme]` CSS 변수 자동 |
| github-markdown-css | 런타임에 light/dark CSS를 동적 import → 이전 link 제거 |
| Prism 토큰 스타일 | 이미 토큰 변수 기반이라 자동 반영 |
| Mermaid 다이어그램 | 테마 변경 시 `mermaid.initialize({theme})` 재호출 + 기존 SVG 재렌더 |
| KaTeX | 색상 variable 기반이라 자동 반영 (수식 색상만 점검) |

### AppConfig 확장

```rust
pub enum ThemePreference { Light, Dark, System }

pub struct AppConfig {
    // ...
    pub theme: ThemePreference,  // default: System
}
```

serde default로 기존 config에 필드 없을 때 `System` 주입.

### 프론트 구현
- `src/lib/stores/theme.svelte.ts`: 현재 테마 계산 (prefers-color-scheme 추적 포함)
- `+layout.svelte`: 초기 config 로드 시 `document.documentElement.dataset.theme = resolved` 설정
- `matchMedia("(prefers-color-scheme)")` change listener — System일 때만 자동 반영
- Settings 페이지에 라디오 3개 (Light / Dark / System)
- 뷰어의 마크다운 파이프라인: 테마 변경 시 현재 노트 재렌더 (pipeline `data-*-rendered` 마커 초기화)

### TDD 테스트 대상
- Rust: AppConfig.theme serde default, serialize/deserialize roundtrip, update_config patch
- 프론트: jsdom 환경에서 테마 store의 OS 변경 구독 동작(향후 프레임워크 도입 시)

### 검증
- 세 옵션(Light/Dark/System) 전환 시:
  - 모든 페이지 배경/텍스트 대비 AA 이상
  - 마크다운 본문 색상 즉시 반영
  - Mermaid/KaTeX 기존 렌더가 새 테마로 교체
- 재시작 후 선택 유지
- 컴팩트 모드와 직교 (Slice 1.2와 교차 검증)

### 위험
- **flash of wrong theme**: 첫 페인트 전에 테마를 정해야 함 → `app.html`에 인라인 스크립트로
  localStorage 값을 읽어 `<html data-theme>` 선반영
- **GitHub CSS 스왑 타이밍**: 동적 import가 느리면 깜빡임 → 두 CSS를 모두 번들에 포함하고
  `data-theme` 셀렉터로 격리하는 방안 검토
- **기존 다크 전용 하드코딩 잔존**: 컴포넌트 grep으로 `bg-surface-*` 외 하드 색상 일괄 검사

### 열린 결정 (Slice 진입 전 확정)
- 라이트 팔레트 베이스: 순백(#FFFFFF) vs 웜화이트(#FAFAF7) → 디자인 콘설팅 후
- 기본값: `System` vs `Dark` (현재 스타일 유지) → 사용자 선호 청취
- Mermaid 기존 SVG 교체 전략: 즉시 재렌더 vs 다음 노트 이동 때 → UX 확인 후
- GitHub CSS 로딩 방식: 동적 import vs 두 파일 모두 번들 + 스코프 격리 → 번들 크기/깜빡임 트레이드오프

## Slice 1.6 — 사이드바 Collapse/Expand

### 배경
Slice 1.4에서 그룹화한 사이드바는 `w-52` 고정. 본문이 좁은 레이아웃(그래프/마크다운 와이드 뷰)에서
공간을 돌려받고 싶을 때가 있다. 토글 방식이 필요.

### 목표
- 사이드바 헤더(또는 footer 근처)에 collapse 토글 버튼
- 접힌 상태: 아이콘 전용 (약 `w-12`), 그룹 라벨 숨김, 볼트 이름/버전 숨김
- 펼친 상태: 현재 `w-52` 그대로
- **AppConfig persist** — 재시작 후 유지
- 전환은 CSS transition(150~200ms)으로 매끄럽게

### AppConfig 확장

```rust
pub struct AppConfig {
    // ...
    #[serde(default)]
    pub sidebar_collapsed: bool,  // default false (펼침)
}
```

### 프론트 구현
- `<html data-sidebar="collapsed">` 속성으로 CSS 분기 (density 패턴과 동일)
- `onMount`에서 초기 config 로드 시 속성 세팅
- 토글 버튼 클릭 → `updateConfig({ sidebar_collapsed })` 즉시 저장 + 속성 갱신
- collapsed 상태에서 각 메뉴 아이템은 아이콘만, `title` 속성으로 라벨 노출(tooltip)
- 볼트 목록: collapsed일 때 활성 볼트의 ● 표시만 남김(또는 섹션 전체 숨김 — 디자인에서 확정)

### TDD 테스트 대상 (Rust)
- `AppConfig.sidebar_collapsed` serde default → false
- `save_then_load_preserves_sidebar_collapsed`
- `merged_with` patch merge

### 검증 (수동)
- 토글 클릭 시 `w-52` ↔ `w-12` 전환 애니메이션
- 접힌 상태에서 각 아이콘 호버 시 tooltip(라벨)
- 재시작 후 상태 유지
- 컴팩트 모드와 교차 — Compact × Collapsed 조합이 자연스러운지
- 라이트/다크 테마와 직교 (Slice 1.5와 교차 검증)

### Open 결정
- 토글 버튼 위치: 헤더 우측 vs footer vs 사이드바 바깥 가장자리 → 디자인 단계에서 확정
- collapsed에서 볼트 섹션 표현: 활성 볼트만 ● / 전체 숨김 / 아이콘 placeholder
- 키보드 단축키: `Cmd/Ctrl + B` 등 → MVP 포함 여부 확인 필요
- 아주 좁은 창(w < 640px)에서 자동 collapse할지 — 반응형 정책은 별도 이슈
