# Jiphyeon v2.0 피벗 로드맵

**작성일**: 2026-04-22
**상태**: 계획 승인 대기

## 배경

v1.0 (Epic 1~4) 완료 직후, `/graphify` 스킬을 실전에서 사용하면서 집현이 자체 구현하던 기능(링크 그래프, God Node, Cluster, 백링크 감사)이 graphify와 상당 부분 중첩됨이 드러났다. graphify는 이미 더 정교한 커뮤니티 감지, God Node 분석, Surprising Connection 탐지, 감사 리포트를 제공한다. 자체 엔진을 유지하는 것은 **두 개의 진실 원점**을 만든다.

v2.0은 집현의 역할을 재정의한다.

## 포지셔닝 전환

| 항목 | v1.0 | v2.0 |
|---|---|---|
| 한 줄 정의 | 마크다운 볼트 대시보드 | Graphify의 데스크톱 프론트엔드 + 멀티프로젝트 퀵캡처 |
| 데이터 엔진 | 자체 인덱서(parser/indexer/search) | graphify-out 읽기 + tantivy full-text |
| 단위 | 볼트 (단일 폴더) | 프로젝트 (레지스트리) |
| 입력 | 볼트 내 노트 | 여러 프로젝트의 docs/ + graphify-out/ |
| 차별점 | 볼트 메타 대시보드 | 크로스 프로젝트 통합 검색 + 크로스 프로젝트 그래프 + 퀵캡처 |

**집현의 세 가지 핵심 기능 + 한 가지 입력 창구:**
1. **검색** — 등록된 모든 프로젝트의 full-text + 그래프 노드 + 과거 Q&A
2. **탐색** — graphify 지식 그래프 시각화 (프로젝트별 / 크로스 프로젝트)
3. **조회** — 마크다운 본문 렌더링 (단일 파일 뷰어)
4. **캡처** — 글로벌 단축키 기반 퀵노트 / URL 클리핑 / 녹음 (활성 프로젝트 `docs/`로 저장)

## 글로벌 / 앱 단축키

| 단축키 | 동작 | 슬라이스 |
|---|---|---|
| Cmd+Shift+N (글로벌) | `/capture` 창 열기 | D-1 |
| Cmd+K | 검색 팔레트 오버레이 | E-4 |
| Cmd+B | 사이드바 collapse/expand | v1.0 유지 |

충돌 방지를 위해 Cmd+N/Cmd+T/Cmd+W 등 OS 예약 단축키는 회피. Cmd+Shift+N은 일부 브라우저와 겹치나 Tauri 앱 포커스 시에만 작동. Cmd+K는 `/capture` 내부에서 저장 등 페이지-로컬 단축키와 네임스페이스 충돌 없도록 주의.

## Epic C. Graphify 뷰어 통합

**목표**: 집현을 graphify-out/ 의 "대시보드 + 그래프 뷰어"로 만든다.

**작업 브랜치**: `feat/v2.0-epic-c-graphify-viewer`

### Rust 모델

```rust
// src-tauri/src/graphify/reader.rs (신규)
pub struct GraphifyNode {
    pub id: String,
    pub label: String,
    pub file_type: String,      // code|document|paper|image
    pub source_file: String,
    pub source_location: Option<String>,
    pub community: Option<i64>,
}

pub struct GraphifyEdge {
    pub source: String,
    pub target: String,
    pub relation: String,
    pub confidence: Confidence,     // EXTRACTED | INFERRED | AMBIGUOUS
    pub confidence_score: f64,
    pub source_file: Option<String>,
    pub weight: f64,
}

pub struct GraphifyHyperedge {
    pub id: String,
    pub label: String,
    pub nodes: Vec<String>,
    pub relation: String,
    pub confidence: Confidence,
    pub confidence_score: f64,
}

pub struct GraphifyGraph {
    pub nodes: Vec<GraphifyNode>,
    pub edges: Vec<GraphifyEdge>,
    pub hyperedges: Vec<GraphifyHyperedge>,
}

pub struct GraphifyReport {
    pub summary: String,              // nodes/edges/communities/cost
    pub god_nodes: Vec<String>,       // 파싱된 God Nodes 섹션
    pub surprising_connections: Vec<String>,
    pub suggested_questions: Vec<String>,
    pub community_labels: HashMap<i64, String>,
}
```

### Slice C-1. graph.json 파서
- Spec: `docs/specs/spec-graphify-reader.md`
- NetworkX `node_link_graph` 형식 파싱 (`nodes`, `links` 키 주의 — `edges`가 아님)
- 파일 없으면 `GraphifyError::NotRun` 반환
- 파일 mtime을 ProjectEntry.last_graphify_at에 기록
- 테스트: 샘플 graph.json 2종 (작은 그래프 / 하이퍼엣지 포함)

### Slice C-2. GRAPH_REPORT.md 파서
- Markdown 섹션 단위 파싱 (heading 기준)
- God Nodes / Surprising Connections / Suggested Questions / Community Hubs 추출
- 테스트: 실측 `bloghub` 리포트 샘플 활용

### Slice C-3. Tauri 커맨드
- `get_graphify_graph(project_id) -> Result<GraphifyGraph, GraphifyError>`
- `get_graphify_report(project_id) -> Result<GraphifyReport, GraphifyError>`
- `get_graphify_status(project_id) -> GraphifyStatus { exists, last_run_at, nodes_count, edges_count, communities_count }`

### Slice C-4. Watcher — graphify-out 감시
- 활성 프로젝트의 `<project>/graphify-out/graph.json` 변경 감시
- 변경 시 `graphify-updated` 이벤트 emit → 프론트 재로드
- 기존 `watcher/`의 notify 인프라 재활용, 필터만 교체

### Slice C-5. `/graph` 페이지 재설계
- `LinkGraph.svelte` 입력 포맷을 GraphifyNode/Edge로 교체
- 커뮤니티별 색상 (d3 ordinal scale, 12색 팔레트 반복)
- 엣지 시각화: `confidence` 기준 — EXTRACTED 실선, INFERRED 얇은 실선, AMBIGUOUS 점선
- `confidence_score` → 엣지 투명도
- 노드 크기: degree 기반 (graphify가 별도 degree 안 주므로 프론트에서 계산)
- 노드 클릭 → (a) 조회 뷰어(`source_file` 열기), (b) 이웃 강조 토글
- 필터: 커뮤니티 선택, relation 타입, confidence 레벨
- 하이퍼엣지: 3+ 노드를 점선 다각형으로 묶음 (2단계 구현, 초기엔 hyperedges 배지만)

### Slice C-6. 대시보드 재설계
- 빈 프로젝트(graphify 미실행): 큰 안내 카드 — "`cd <project-root> && /graphify` 를 실행하세요"
- graphify 실행된 프로젝트:
  - Summary 카드: 노드/엣지/커뮤니티 수 + 마지막 실행 시각 + 토큰 비용
  - God Nodes 카드 (GRAPH_REPORT의 God Nodes 섹션)
  - Surprising Connections 카드
  - Suggested Questions 카드 — 각 질문 클릭 시 `/graph`에서 관련 노드 포커스
  - Communities 카드: community_labels + size 내림차순 상위 5개

### Slice C-7. 크로스 프로젝트 그래프 뷰
**집현 고유의 차별점**. 여러 프로젝트의 graph.json을 합쳐서 개념 연결을 시각화.

- 진입점: `/graph?scope=cross` 또는 `/graph` 상단 "전체 프로젝트" 토글
- UI: 프로젝트 멀티 셀렉트 체크박스 (기본값 = 등록된 모든 프로젝트)
- Rust 커맨드: `get_cross_project_graph(project_ids: Vec<String>, merge_labels: bool) -> GraphifyGraph`
- **머지 알고리즘:**
  1. 각 프로젝트의 graph.json 로드
  2. 노드 ID 네임스페이싱: `{project_id}::{node_id}` (충돌 방지)
  3. 노드 메타에 `project_id` 필드 추가 (시각화에서 프로젝트별 색상에 활용)
  4. 엣지도 동일하게 네임스페이싱
  5. **동일 label 병합 옵션(merge_labels=true)**: `norm_label` 기준으로 동일 라벨 노드들을 가상 브리지 노드로 연결 (`cross_project_alias` relation, confidence=INFERRED, score=0.5)
     - 예: 프로젝트 A의 `"JWT Validator"`와 프로젝트 B의 `"JWT 검증"`이 둘 다 `jwt validator` normalized → 브리지 엣지로 연결
     - 시각화: 브리지 엣지는 굵은 점선 + 별도 색상
- 시각화 (`LinkGraph.svelte` 확장):
  - 노드 색상: 프로젝트별 해시 팔레트 (community 색상과 2단계로 구분 — outer ring = project, inner fill = community)
  - 프로젝트 범례 오버레이
  - 브리지 엣지 on/off 토글
  - 대규모 시(>3000 노드) 경고 + 프로젝트 필터링 권유
- 탐색 액션:
  - 노드 클릭 → 해당 프로젝트로 switch + 단일 프로젝트 그래프 모드로 포커스
  - 브리지 엣지 클릭 → "두 프로젝트에서 같은 개념" 패널 (양쪽 source_file 보여줌)

**Slice C 완료 조건**: 프로젝트 하나 등록 + `/graphify` 실행 → 집현에서 그래프/대시보드 정상 표시. 2개 이상 프로젝트 등록 시 크로스 프로젝트 모드에서 브리지 엣지 표시. graphify 미실행 프로젝트는 온보딩 카드.

## Epic D. Capture 통합 (Quicknote / URL / 녹음)

**목표**: `/capture` 단일 창에서 텍스트 노트 / URL 클리핑 / 녹음 세 가지 입력을 모두 처리. 활성 프로젝트 `docs/` 하위에 저장.

**작업 브랜치**: `feat/v2.0-epic-d-capture`

### 저장 경로 규약

| 타입 | 저장 경로 |
|---|---|
| 퀵노트 | `<project>/docs/inbox/YYYY-MM-DD-{slug}.md` |
| URL 클리핑 | `<project>/docs/clippings/YYYY-MM-DD-{slug}.md` |
| 녹음 | `<project>/docs/_sources/recordings/YYYY-MM-DD-HHMMSS.{ext}` |

`_sources/` prefix는 graphify에 "원본 미디어"로 인식됨 (Step 2에서 video로 detect → Step 2.5에서 Whisper 전사).

### Slice D-1. `/capture` 창 재설계
- 상단: 활성 프로젝트 셀렉터 (기본값 = 마지막 활성 프로젝트, 드롭다운으로 오버라이드)
- 탭 3개: 📝 노트 · 🔗 URL · 🎙️ 녹음
- 공통 frontmatter 자동:
  ```yaml
  ---
  created: 2026-04-22T14:33:00+09:00
  tags: []
  source: capture
  ---
  ```
- 저장 후 토스트: "저장됨. 그래프 갱신하려면 `/graphify --update` 실행"
- Cmd+Shift+N 글로벌 단축키 유지

### Slice D-2. 퀵노트 탭
- 제목 입력 + 본문 마크다운 (monospace)
- slug = 제목을 kebab-case로 변환
- 저장 커맨드: `save_quick_note(project_id, title, content)`

### Slice D-3. URL 클리핑 탭
- 기존 `commands/clipper.rs::clip_url` 재활용
- 저장 경로만 활성 프로젝트의 `docs/clippings/`로 변경
- 프론트매터에 `source_url`, `captured_at` 추가 (graphify가 Step 2 frontmatter 로직으로 source_url을 노드 메타에 반영함)

### Slice D-4. 녹음 탭
- 기존 `src/lib/transcribe/` 유틸 재활용 (MediaRecorder, 포맷 폴백, Canvas 파형, 60분 rolling)
- 저장 커맨드: `save_recording(project_id, filename, bytes)` — 기존 로직을 활성 프로젝트 경로 기준으로
- 전사 상태 뱃지 제거 (graphify가 `_sources/recordings/` 스캔 시 자동 전사)
- macOS 마이크 권한 관련 Info.plist 설정 유지

### Slice D-5. 기존 `/transcribe` 페이지 제거
- `src/routes/transcribe/+page.svelte` 삭제
- 사이드바 `/transcribe` 메뉴 제거 (이미 Epic A에서 제거했으면 skip)

### Slice D-6. "Pending graphify" 뱃지
- 활성 프로젝트 사이드바에 상태 뱃지:
  - ✅ `graph.json mtime > docs/ 최신 mtime` — 최신
  - ⚠️ `graph.json mtime < docs/ 최신 mtime` — "N개 파일 변경. graphify 재실행 필요"
  - ❌ `graph.json 없음` — "graphify 미실행"
- docs/ 최신 mtime은 Epic C-4 watcher에서 같이 감지 (graphify-out뿐 아니라 docs/도 감시)

**Slice D 완료 조건**: Cmd+Shift+N → 퀵노트 저장 → 파일 생성 확인. 녹음 저장 후 `/graphify --update` 돌렸을 때 전사본이 graph에 나타남.

## Epic E. 통합 검색 (E-a+ Hybrid)

**목표**: 모든 등록된 프로젝트의 문서 본문 + 그래프 노드 + 과거 Q&A를 단일 쿼리로 검색.

**작업 브랜치**: `feat/v2.0-epic-e-search`

### tantivy 스키마 재설계

```rust
// src-tauri/src/search/schema.rs
// 단일 인덱스, 문서 타입을 "kind" 필드로 구분
struct SearchDoc {
    project_id: String,       // STRING (facet)
    kind: String,             // STRING: "file" | "node" | "qa"
    // 공통
    title: String,            // TEXT (analyzer: korean-english)
    body: String,             // TEXT
    path: String,             // STRING (파일 경로 or node id)
    // file-only
    file_ext: Option<String>, // STRING
    // node-only
    node_id: Option<String>,  // STRING
    community: Option<i64>,   // INT
    relation: Option<String>, // STRING (엣지 타겟 요약)
}
```

### Slice E-1. 인덱서 재구현
- Spec: `docs/specs/spec-search-index.md`
- 인덱싱 소스 (프로젝트 단위):
  1. `graph.json` → node 단위로 인덱싱 (kind=node)
  2. `graph.json`의 `node.source_file` 집합 → 해당 파일 읽어서 full-text 인덱싱 (kind=file)
  3. `GRAPH_REPORT.md` → kind=file (특별 취급)
  4. `graphify-out/memory/query_*.md` → kind=qa
- 삭제 처리: graph.json 재파싱 시 해당 프로젝트의 기존 문서를 전체 삭제 후 재삽입 (단순화)
- 트리거:
  - 프로젝트 등록 시 초기 인덱싱
  - graphify-updated 이벤트 시 재인덱싱 (Epic C-4 watcher 재활용)

### Slice E-2. 검색 커맨드
- `search(query, project_filter: Option<Vec<String>>, kind_filter: Option<Vec<String>>) -> Vec<SearchHit>`
- SearchHit:
  ```rust
  struct SearchHit {
      project_id: String,
      project_name: String,
      kind: String,
      title: String,
      snippet: String,       // 하이라이트된 발췌
      path: String,
      score: f32,
  }
  ```
- 결과 정렬: 기본은 BM25 score. 옵션으로 프로젝트별 그룹핑.

### Slice E-3. 검색 UI (`/search` 페이지)
- 상단 검색창 + 프로젝트 필터 칩 + kind 필터 (전체/파일/노드/Q&A)
- 결과 카드:
  - 📁 프로젝트명 배지
  - 타입 아이콘 (📄 file, 🧩 node, 💬 qa)
  - 제목 + 스니펫 (하이라이트)
  - 경로 + 액션 버튼 (조회 / 그래프에서 보기)
- 클릭 동작:
  - file → 해당 프로젝트로 switch + `/view?path=...`
  - node → 해당 프로젝트로 switch + `/graph?focus=<node_id>`
  - qa → 조회 뷰어로 memory 파일 렌더

### Slice E-4. 전역 검색 단축키
- Cmd+K (혹은 Cmd+P) → 인라인 검색 팔레트 오버레이
- 퀵노트(Cmd+Shift+N)와 다른 단축키
- 상위 5~10개 결과 즉시 표시, Enter로 이동

**Slice E 완료 조건**: 3개 프로젝트 등록 + 각각 graphify 실행 → "JWT" 같은 쿼리로 프로젝트를 가로지르는 결과 표시.
